\section{Introducci\'on}

La presente memoria documenta el trabajo realizado en la Pr\'actica 04 de Programaci\'on en Internet sobre el caso de estudio \textbf{SubSonic Festival}. Tras completar previamente el desarrollo del back-end, la integraci\'on con el front-end y la validaci\'on de los perfiles Visitante, Cliente y Proveedor, esta entrega se centra en preparar el proyecto para su ejecuci\'on mediante contenedores Docker y su despliegue en Microsoft Azure.

El resultado final es una aplicaci\'on web desplegable en la nube, empaquetada mediante Docker y publicada en Azure Container Apps. La aplicaci\'on mantiene el front-end en HTML, CSS y JavaScript puro, y el back-end en Python con FastAPI. FastAPI sirve tanto la API REST como las vistas est\'aticas del proyecto, lo que permite desplegar todo el sistema como una \'unica imagen de contenedor.

Junto al desarrollo t\'ecnico, esta memoria recoge la estructura Docker incorporada al repositorio, el proceso de construcci\'on de la imagen, la publicaci\'on mediante GitHub Actions, el despliegue final en Azure, las comprobaciones realizadas y las actas de seguimiento asociadas a la pr\'actica.

\section{Objetivos y alcance de la pr\'actica}

De acuerdo con el enunciado oficial, la pr\'actica exige crear una estructura Docker para el proyecto, incluir los archivos necesarios dentro del repositorio, construir la imagen sin errores y desplegar la aplicaci\'on en un proveedor cloud. En nuestro caso, el proveedor utilizado ha sido \textbf{Microsoft Azure}.

El alcance alcanzado cubre los siguientes puntos:

\begin{itemize}
  \item Incorporaci\'on de una carpeta \texttt{Docker} al repositorio.
  \item Creaci\'on de un \texttt{Dockerfile} funcional para ejecutar la aplicaci\'on.
  \item Definici\'on de un archivo \texttt{.dockerignore} para optimizar el contexto de construcci\'on.
  \item Construcci\'on y validaci\'on de la imagen Docker.
  \item Publicaci\'on autom\'atica de la imagen mediante GitHub Actions.
  \item Almacenamiento de la imagen en GitHub Container Registry.
  \item Despliegue de la aplicaci\'on en Azure Container Apps.
  \item Obtenci\'on de una URL p\'ublica de acceso.
  \item Documentaci\'on del proceso seguido y de las pruebas realizadas.
\end{itemize}

Durante el proceso se intent\'o utilizar Azure Container Registry como registro de im\'agenes, pero la suscripci\'on de Azure for Students utilizada bloque\'o la creaci\'on de dicho recurso mediante una pol\'itica de regiones/recursos. Por este motivo, se decidi\'o utilizar GitHub Container Registry como registro externo de la imagen Docker y Azure Container Apps como servicio final de despliegue.

\section{Arquitectura del sistema y organizaci\'on del proyecto}

\subsection{Tecnolog\'ias empleadas}

El proyecto mantiene las tecnolog\'ias utilizadas en las entregas anteriores. El front-end est\'a desarrollado con HTML, CSS y JavaScript puro, sin frameworks adicionales. El back-end se ha desarrollado en Python mediante FastAPI, que expone la API REST y sirve tambi\'en los recursos est\'aticos del front-end.

Para esta pr\'actica se han incorporado las siguientes tecnolog\'ias de despliegue:

\begin{itemize}
  \item \textbf{Docker}, para empaquetar la aplicaci\'on y sus dependencias.
  \item \textbf{GitHub Actions}, para construir y publicar la imagen de forma autom\'atica.
  \item \textbf{GitHub Container Registry}, para almacenar la imagen Docker.
  \item \textbf{Azure Container Apps}, para ejecutar el contenedor en la nube y exponerlo mediante una URL p\'ublica.
\end{itemize}

\subsection{Estructura relevante del repositorio}

La estructura final mantiene la organizaci\'on previa del proyecto y a\~nade los elementos necesarios para la dockerizaci\'on:

\begin{itemize}
  \item \texttt{backend/}: implementaci\'on del servidor FastAPI, controladores, modelo, DAO, DTO y datos JSON.
  \item \texttt{pages/}: vistas HTML del front-end.
  \item \texttt{js/}: scripts JavaScript que conectan el front-end con la API.
  \item \texttt{css/}: hojas de estilo.
  \item \texttt{Docker/Dockerfile}: definici\'on de la imagen Docker.
  \item \texttt{Docker/README.md}: instrucciones de construcci\'on y prueba.
  \item \texttt{Docker/azure-deploy-template.ps1}: plantilla auxiliar para despliegue con Azure Container Registry.
  \item \texttt{.dockerignore}: exclusiones del contexto Docker.
  \item \texttt{.github/workflows/docker-ghcr.yml}: workflow para construir y publicar la imagen.
  \item \texttt{docs/practica04/}: memoria final de la pr\'actica.
\end{itemize}

\subsection{Relaci\'on con la arquitectura previa}

La dockerizaci\'on no modifica la arquitectura funcional del proyecto. Se mantiene la separaci\'on entre Controller, Model, DAO, Factory y DTO, as\'i como la persistencia local mediante archivos JSON. Docker act\'ua como una capa de empaquetado y despliegue, pero no altera la l\'ogica de negocio ni la integraci\'on entre front-end y back-end.

Esta decisi\'on permite que la misma aplicaci\'on que funcionaba en localhost pueda ejecutarse ahora dentro de un contenedor y, posteriormente, en Azure, manteniendo el comportamiento ya validado en la Pr\'actica 03.

\section{Dockerizaci\'on del proyecto}

\subsection{Imagen base utilizada}

La imagen base utilizada en el \texttt{Dockerfile} es:

\begin{verbatim}
python:3.11-slim
\end{verbatim}

Esta imagen procede de Docker Hub y se utiliza como base ligera para ejecutar aplicaciones Python. Sobre ella se instalan las dependencias del proyecto definidas en \texttt{requirements.txt}.

\subsection{Contenido del Dockerfile}

El \texttt{Dockerfile} realiza las siguientes acciones:

\begin{enumerate}
  \item Usa \texttt{python:3.11-slim} como imagen base.
  \item Define \texttt{/app} como directorio de trabajo.
  \item Copia \texttt{requirements.txt}.
  \item Instala las dependencias necesarias.
  \item Copia las carpetas \texttt{backend}, \texttt{pages}, \texttt{js} y \texttt{css}.
  \item Copia \texttt{smoke\_test.py} para permitir comprobaciones dentro del contenedor.
  \item Expone el puerto 8000.
  \item Arranca FastAPI mediante Uvicorn escuchando en \texttt{0.0.0.0}.
\end{enumerate}

El comando de arranque de la imagen es:

\begin{verbatim}
python -m uvicorn backend.controller.controller:app --host 0.0.0.0 --port 8000
\end{verbatim}

Se utiliza \texttt{0.0.0.0} porque dentro de un contenedor la aplicaci\'on debe aceptar conexiones desde fuera del propio contenedor.

\subsection{Construcci\'on local de la imagen}

La imagen puede construirse desde la ra\'iz del repositorio con:

\begin{verbatim}
docker build -f Docker/Dockerfile -t subsonic-festival:practica04 .
\end{verbatim}

Y puede ejecutarse localmente mediante:

\begin{verbatim}
docker run --rm -p 8000:8000 subsonic-festival:practica04
\end{verbatim}

Una vez iniciado el contenedor, la aplicaci\'on queda disponible en:

\begin{itemize}
  \item \url{http://127.0.0.1:8000/pages/visitante.html}
  \item \url{http://127.0.0.1:8000/pages/login.html}
  \item \url{http://127.0.0.1:8000/docs}
\end{itemize}

\section{Publicaci\'on de la imagen Docker}

\subsection{Uso de GitHub Actions}

Para evitar depender de una construcci\'on manual de la imagen, se ha a\~nadido un workflow de GitHub Actions:

\begin{verbatim}
.github/workflows/docker-ghcr.yml
\end{verbatim}

Este workflow se ejecuta al subir cambios a la rama \texttt{practica04-docker-azure} y realiza el siguiente flujo:

\begin{enumerate}
  \item Descarga el repositorio.
  \item Inicia sesi\'on en GitHub Container Registry mediante \texttt{GITHUB\_TOKEN}.
  \item Construye la imagen usando \texttt{Docker/Dockerfile}.
  \item Publica la imagen con la etiqueta \texttt{practica04}.
\end{enumerate}

\subsection{Registro de contenedor utilizado}

La imagen final publicada es:

\begin{verbatim}
ghcr.io/subsonic-grupo4/l4g1/subsonic-festival:practica04
\end{verbatim}

Esta imagen contiene la aplicaci\'on completa SubSonic Festival, incluyendo el back-end FastAPI, las vistas HTML, los scripts JavaScript, los estilos CSS y los archivos JSON necesarios para la persistencia local.

\section{Despliegue en Microsoft Azure}

\subsection{Servicio utilizado}

El servicio cloud utilizado para la ejecuci\'on final es \textbf{Azure Container Apps}. Esta opci\'on permite ejecutar aplicaciones basadas en contenedores sin tener que administrar manualmente una m\'aquina virtual, el sistema operativo o el servidor web.

La aplicaci\'on se despleg\'o creando una nueva Container App desde el portal de Azure y seleccionando como origen de imagen \textbf{Docker Hub u otros registros}. Al tratarse de una imagen almacenada en GitHub Container Registry, se indic\'o \texttt{ghcr.io} como servidor de registro.

\subsection{Configuraci\'on aplicada en Azure}

La configuraci\'on principal de Azure fue:

\begin{itemize}
  \item Grupo de recursos: \texttt{l4g1-subsonic-rg}.
  \item Nombre de la aplicaci\'on: \texttt{subsonic-app}.
  \item Regi\'on: Spain Central.
  \item Servicio: Azure Container Apps.
  \item Servidor de registro: \texttt{ghcr.io}.
  \item Imagen: \texttt{ghcr.io/subsonic-grupo4/l4g1/subsonic-festival:practica04}.
  \item Tipo de imagen: privada, utilizando token de GitHub con permisos de lectura de paquetes.
  \item Entrada externa: habilitada.
  \item Tipo de entrada: HTTP.
  \item Puerto de destino: 8000.
\end{itemize}

\section{Validaci\'on y pruebas}

\subsection{Validaci\'on local}

Antes del despliegue en Azure se comprob\'o el funcionamiento del proyecto en local mediante el script:

\begin{verbatim}
python smoke_test.py
\end{verbatim}

El resultado esperado y obtenido fue:

\begin{verbatim}
SMOKE TEST OK
\end{verbatim}

Este script valida acceso p\'ublico a eventos, login de Cliente y Proveedor, rutas protegidas, control de autorizaci\'on por rol y funcionamiento b\'asico de la API.

\subsection{Validaci\'on de Docker}

Tambi\'en se comprob\'o que la imagen Docker se constru\'ia correctamente y que el contenedor expon\'ia las rutas principales:

\begin{itemize}
  \item \texttt{/pages/visitante.html}
  \item \texttt{/pages/login.html}
  \item \texttt{/docs}
  \item \texttt{/events}
\end{itemize}

Adem\'as, se valid\'o la ejecuci\'on del test autom\'atico dentro del contenedor.

\subsection{Validaci\'on en Azure}

Tras el despliegue en Azure Container Apps se verific\'o que la URL p\'ublica resolv\'ia correctamente y que las rutas principales del sistema pod\'ian abrirse desde navegador. Esta comprobaci\'on confirma que la aplicaci\'on no solo funciona en localhost, sino tambi\'en en un entorno cloud real.

\section{Estado final del proyecto}

La versi\'on final entregable cumple los objetivos de la Pr\'actica 04. El repositorio contiene la estructura Docker solicitada, la imagen puede construirse a partir del \texttt{Dockerfile}, el proceso de publicaci\'on se automatiza mediante GitHub Actions y la aplicaci\'on est\'a desplegada en Azure Container Apps con URL p\'ublica.

El proyecto conserva adem\'as todo el trabajo desarrollado en la Pr\'actica 03: perfiles Visitante, Cliente y Proveedor, API REST con FastAPI, autenticaci\'on por token, arquitectura MVC + DAO + Factory + DTO y persistencia local mediante JSON. La pr\'actica actual no sustituye esa funcionalidad, sino que la empaqueta y la lleva a un entorno de despliegue cloud.

Por tanto, la rama \texttt{practica04-docker-azure} puede considerarse la versi\'on final del proyecto para la revisi\'on docente, ya que integra tanto la aplicaci\'on funcional como el despliegue mediante Docker y Azure.

Para facilitar la revisi\'on docente, se proponen dos formas de comprobar el proyecto: una comprobaci\'on directa desde Azure y una comprobaci\'on local mediante Docker.

\subsection{Opci\'on 1: comprobaci\'on directa en Azure}

La forma m\'as r\'apida de probar la aplicaci\'on es acceder a la URL p\'ublica del despliegue:

\begin{verbatim}
https://subsonic-app.gentlefield-85aec19e.spaincentral.azurecontainerapps.io
\end{verbatim}

Rutas recomendadas:

\begin{itemize}
  \item Perfil Visitante: \url{https://subsonic-app.gentlefield-85aec19e.spaincentral.azurecontainerapps.io/pages/visitante.html}
  \item Login: \url{https://subsonic-app.gentlefield-85aec19e.spaincentral.azurecontainerapps.io/pages/login.html}
  \item Documentaci\'on Swagger: \url{https://subsonic-app.gentlefield-85aec19e.spaincentral.azurecontainerapps.io/docs}
\end{itemize}

Credenciales de prueba:

\begin{itemize}
  \item Cliente: \texttt{cliente@subsonic.es} / \texttt{password123}
  \item Proveedor: \texttt{proveedor@subsonic.es} / \texttt{password123}
  \item Admin/proveedor: \texttt{admin@subsonic.es} / \texttt{admin123}
\end{itemize}

Recorrido recomendado:

\begin{enumerate}
  \item Abrir la vista de Visitante y comprobar que se muestran eventos p\'ublicos.

  \item Iniciar sesi\'on como Cliente y comprobar tickets, detalle, compra, cancelaci\'on y perfil.

El contenedor ejecuta FastAPI mediante Uvicorn en el puerto 8000 y sirve tanto la API REST como las p\'aginas del front-end.
